06-spec-manifest-reconciliation

Sequence 06 · Feature: manifest-reconciliation · Phase 1

Introduction/Overview

Today, when a workspace manifest diverges between machines, DevSpace sync refuses and stops: the git path returns "remote branch diverged … pull/reconcile manually" / "remote manifest is newer" (workspace_sync.go), and the hosted path returns an HTTP 409 "version conflict" (hosted_sync.go). The user is left to reconcile by hand, and there are no force flags and no merge tooling. workspace diff can show what a pull would change (a ManifestDiff of Added/Removed/Changed projects), but nothing combines the two sides.

This feature adds a reconciliation path: a new devspace workspace reconcile command that performs a three-way, project-level merge of the local and remote manifests against a tracked base (last-synced) manifest. Non-conflicting changes (each side added/removed/changed different projects) merge automatically; genuine conflicts (the same project changed differently on both sides) are never auto-resolved — they are surfaced for an explicit decision. Reconciliation is non-destructive and review-first: it writes a reviewable reconcile plan (mirroring plan→apply) and backs up the pre-reconcile manifest before writing anything. It covers both sync backends (git remote and hosted control-plane), sharing one backend-agnostic merge engine.

Goals

  • Turn manifest divergence from a dead-end error into a reviewable, non-destructive reconciliation for both the git and hosted backends.
  • Auto-merge non-conflicting project add/remove/change operations using a tracked base manifest; never lose a tracked project silently.
  • Always surface genuine conflicts (same project changed on both sides) and require an explicit choice — never auto-pick a winner by default.
  • Preserve DevSpace's "review then act" and non-destructive invariants: a reconcile plan is written and the local manifest is backed up before any overwrite; apply is hash-guarded.
  • Reuse the existing Manifest/Project/ManifestDiff types and localize-remote machinery; add no new dependency.

User Stories

  • As a developer syncing a workspace across two machines, I want to merge divergent manifests instead of hitting a "diverged, fix manually" error, so that both machines' project additions survive without hand-editing JSON.
  • As a developer, I want genuine conflicts (a project changed differently on each machine) flagged for me to decide, so that reconciliation never silently discards one side's change.
  • As a cautious user, I want to review the reconciled result and have my previous manifest backed up before anything is written, so that I can roll back if the merge isn't what I expected.
  • As a hosted-sync user, I want the same reconciliation when a push is rejected with a version conflict, so that the control-plane path isn't a second dead end.

Demoable Units of Work

Three vertical slices in build order: track the base → merge engine + git reconcile command → hosted reconcile. The merge engine is written once against the shared Manifest types and reused by both backends (Q3 = both).

Unit 1: Track the last-synced base manifest

Purpose: Record the manifest as of the last successful sync — the base that makes real three-way conflict detection possible (Q2). Without it, "changed on both sides" cannot be told apart from "independently added on both sides."

Functional Requirements:

  • The system shall persist the successfully-synced manifest to a base snapshot in app home (e.g. ~/.devspace/last-synced-manifest.json) after every successful workspace pull/push and hosted pull/push, written atomically with the same permissions as other app-home state.
  • The system shall expose a loader that returns the base manifest (and a clear "no base recorded yet" signal for first-run/upgrade cases).
  • The base snapshot shall contain manifest metadata only (no secrets, no machine-local paths) — consistent with what sync already shares.

Proof Artifacts:

  • Test: go test ./internal/devspace -run TestSyncRecordsBaseManifest passes — a successful git pull/push writes a base snapshot equal to the synced manifest (temp DEVSPACE_HOME).
  • Test: go test ./internal/devspace -run TestBaseManifestAbsentIsDetectable passes — the loader reports "no base" cleanly when none exists.

Unit 2: Three-way merge engine + workspace reconcile (git)

Purpose: The core hybrid merge (Q1=D) plus the review-first command (Q4=A) for the git backend.

Functional Requirements:

  • The system shall provide a backend-agnostic merge function Reconcile(base, local, remote Manifest) ReconcileResult keyed by project path: entries added/removed/changed on only one side (relative to base) auto-merge; an entry changed differently on both sides (or add/add with different content) is recorded as a conflict. It shall never auto-resolve a conflict.
  • The system shall add devspace workspace reconcile that localizes the remote manifest (reusing the existing workspace diff/pull machinery), loads the base (Unit 1), runs the merge, and writes a reviewable reconcile-plan artifact (auto-merged ops + flagged conflicts), plus a --json form consistent with diff/plan.
  • When conflicts exist, the command shall stop without writing the manifest and report each conflict with text labels; it shall resolve them only when the user supplies an explicit choice (--force-local / --force-remote).
  • On apply/confirm the system shall back up the pre-reconcile local manifest to app home, then write the reconciled manifest through safeWorkspacePath-checked, atomic writes, guarded by a manifest-hash check (mirroring apply). The whole operation shall run under withAppLock at the command boundary.
  • If no base is recorded, the system shall degrade to a documented two-way mode (treat every same-path difference as a conflict) with a clear notice, never a silent overwrite.

Proof Artifacts:

  • Test: go test ./internal/devspace -run TestReconcileMergeCases passes — table test covering add-only-one-side, remove-one-side, change-one-side (auto-merge) and change-both-sides / add-add-different (conflict), plus the no-base two-way fallback.
  • Test: go test ./internal/devspace -run TestWorkspaceReconcileNonConflicting passes — reconcile of two divergent-but-mergeable manifests produces the union, backs up the old manifest, and applies under the hash guard (temp workspace).
  • Test: go test ./internal/devspace -run TestWorkspaceReconcileConflictBlocksApply passes — a genuine conflict stops apply; --force-local/--force-remote resolves it explicitly.
  • CLI: devspace workspace reconcile --json output demonstrates the auto-merged ops + conflict list.

Unit 3: Hosted-backend reconciliation

Purpose: Reuse the Unit 2 engine on the hosted control-plane path so an HTTP 409 version conflict becomes reconcilable (Q3 = both backends).

Functional Requirements:

  • The system shall let reconciliation target the hosted backend: on a hosted version conflict, fetch the current hosted manifest + version/hash, run the same Reconcile against the tracked base, and present the same review artifact.
  • On apply, the system shall push the reconciled manifest with the correct updated version/hash baseline (so the push is accepted), then update the base snapshot (Unit 1).
  • Conflict handling, backup, and review-before-write shall behave identically to the git path (shared engine + shared reconcile-plan artifact).

Proof Artifacts:

  • Test: go test ./internal/devspace -run TestHostedReconcileResolvesVersionConflict passes — a simulated 409 → reconcile → apply pushes the merged manifest with the updated baseline (using the in-process hosted server test harness).
  • Test: go test ./internal/devspace -run TestHostedReconcileConflictBlocksPush passes — a genuine conflict blocks the hosted push until an explicit force choice is given.

Non-Goals (Out of Scope)

  1. Project content merging: reconciliation operates on the manifest's project entries only — it never merges files inside a project or touches project git repos.
  2. Automatic conflict resolution: DevSpace never silently picks a winner for a genuine conflict; --force-local/--force-remote are explicit opt-ins only (Q2=A).
  3. Text/line-level JSON merge: merging is entry/field-level keyed by project path, not a textual 3-way of the JSON.
  4. Local Config/State reconciliation: machine-local config.json/state.json are not synced and not reconciled — only the shared manifest.json.
  5. Changing default pull/push: plain pull/push keep their current refuse-on-divergence behavior; reconcile is the explicit opt-in path.
  6. Multi-remote base tracking: a single last-synced base is assumed (matches the current single-remote model) — see Open Questions.

Design Considerations

  • Review-first artifact: reconcile writes a reconcile-plan (auto-merged ops + conflicts) analogous to last-plan.json, with a --json form matching diff/plan/doctor output conventions; conflicts render with text labels (not color alone).
  • Backup + rollback: the pre-reconcile local manifest is copied to app home before any write, giving a concrete rollback point.
  • Command placement: under the existing workspace command group (devspace workspace reconcile); hosted reconciliation reachable via the same command targeting the hosted backend (flag or subcommand, at implementation's discretion — see Open Questions).
  • No specific visual/UI requirements beyond CLI output consistency.

Repository Standards

  • Single package internal/devspace; register the command via NewRootCommand's AddCommand pattern under the workspace group.
  • Reuse Manifest/Project/ManifestDiff, the localize-remote-manifest machinery behind workspace diff/pull, atomic JSON writes (jsonio), and safeWorkspacePath; never hand-join workspace paths.
  • Lock discipline: acquire withAppLock only at the command boundary (non-reentrant), per lock.go.
  • Non-destructive + idempotent: backup before write; hash-guarded apply; re-running reconcile on an already-merged state is a no-op.
  • Testing: t.Setenv("DEVSPACE_HOME", tmp) + temp workspace; behavior-named *_test.go; reuse the in-process hosted server test harness for Unit 3; gate with make verify (test + vet + lint incl. gosec + govulncheck + build).
  • Conventional Commits; document the new command in README.md "Supported Commands".

Technical Considerations

Base manifest (load-bearing — Q2): a new app-home snapshot (e.g. last-synced-manifest.json) written atomically on every successful pull/push for both backends. It is the enabling mechanism for true three-way detection; the hosted backend already carries a version/hash baseline, but three-way still needs the base content, so both backends cache it the same way. First-run/upgrade with no base → documented two-way fallback (all same-path differences are conflicts) with a clear notice; never a silent overwrite.

Merge algorithm (Q1=D): pure function Reconcile(base, local, remote) keyed by project Path. For each path, classify each side vs base (unchanged / added / removed / changed); combine: one-sided changes auto-apply; both-sided-and-different (change/change, add/add-different, or change/remove) → conflict. Output a ReconcileResult {merged ops, conflicts[]}. Deterministic and unit-testable in isolation — this is where the bulk of the test coverage lives.

Two backends (Q3=C): the engine is backend-agnostic. Git wiring: localize remote via the existing pull path; apply writes the local manifest and (optionally) pushes. Hosted wiring: on 409, fetch the current hosted manifest + version/hash; merge; apply pushes with the new baseline. Both update the base snapshot on success. Keep the engine and the two wirings cleanly separated so hosted reuse is thin.

Concurrency: reconcile + apply hold withAppLock for the operation (non-reentrant); domain reads/writes assume the caller holds the lock. Writes are atomic (existing jsonio pattern).

Security Considerations

  • Reconciliation touches manifest.json only — no secrets, no project contents, no machine-local paths (sync already strips those).
  • The base snapshot and the pre-reconcile backup are manifest metadata written under app home with the same permissions as existing state files.
  • Hosted reconciliation uses the existing bearer-token auth (DEVSPACE_HOSTED_TOKEN); no new credentials or scopes.
  • Proof artifacts must use sanitized temp workspaces / the in-process test server — no real remotes, tokens, or secret values committed.

Success Metrics

  1. Non-conflicting merges are automatic: two manifests that added/removed/changed different projects reconcile with zero prompts and no lost projects.
  2. Conflicts are never silently resolved: a genuine conflict always stops apply and requires an explicit force choice — verified by test.
  3. Base is always recorded: every successful pull/push (git and hosted) writes the base snapshot.
  4. Non-destructive: pre-reconcile manifest is backed up and apply is hash-guarded, in 100% of runs.
  5. Both backends covered: git and hosted reconciliation share the engine and both pass their proof tests.
  6. Gate green: make verify passes with new engine + CLI + hosted tests.

Open Questions

Non-blocking; resolvable during implementation without changing scope, requirements, demoable units, or proof artifacts:

  1. Conflict-resolution granularity: start with global --force-local/--force-remote, or support per-project selection immediately? Global is the simpler first cut; per-project can follow. Non-blocking.
  2. Hosted invocation surface: a flag on workspace reconcile vs a dedicated hosted reconcile subcommand — either satisfies the requirement; implementation picks the cleaner fit with the existing command groups.
  3. Base keying if multiple remotes are ever supported (single snapshot vs per-remote) — single for now, matching the current single-remote model.
