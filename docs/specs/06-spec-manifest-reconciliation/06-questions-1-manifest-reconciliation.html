Feature context. Today, manifest sync refuses and stops on divergence: workspace_sync.go returns "remote branch diverged … pull/reconcile manually" and "remote manifest is newer" (lines 365/399/401); hosted sync returns HTTP 409 "version conflict" (hosted_sync.go 309/813). workspace diff already reports what a pull would change as a ManifestDiff (Added/Removed/Changed project lists). There are no force flags and no reconciliation path — the user is left to fix it by hand. Spec 06 adds a reconciliation path. The four questions below are the material decisions; the implementation will be routed to Codex per the wave-5 execution decision.

1. Reconciliation / Merge Strategy

When two manifests diverge, how should DevSpace combine them?

Current best-practice context: The manifest is structured JSON keyed by relative project path, so a field/entry-level three-way merge is tractable and far safer than a line/text merge. A three-way merge needs a tracked base (the last-synced manifest) to tell "added on both" apart from "changed on both" — see Q2.
Recommended answer(s): (D)

Why these are recommended:

  • (D) resolves the common, safe case (each machine added/removed different projects) with zero prompts, while never silently picking a winner for a true conflict — matching DevSpace's non-destructive-by-default posture.
  • (B) alone is noisy (prompts even for trivially-mergeable divergence); (C) alone is a blunt instrument that discards one side's work; (A) alone leaves true conflicts undefined.
  • (D) still allows (C)-style force flags as an escape hatch (see Q4) without making overwrite the default.

2. Conflict Authority + Common-Ancestor Base

For a genuine conflict (same project path present with different content on both sides), what is the default, and do we track a base manifest to detect true conflicts?

Current best-practice context: Without a stored base, "both sides have project X but different" is indistinguishable from "both independently added X" — so a safe merge needs a base snapshot (the manifest as of the last successful pull/push). The hosted path already carries a version/hash baseline; the git path would need to cache the last-synced manifest locally (app home).
Recommended answer(s): (A)

Why these are recommended:

  • (A) keeps the invariant that DevSpace never destroys user data without explicit permission; a wrong auto-pick silently drops a tracked project.
  • Tracking a base manifest is the enabling mechanism for Q1(D); without it, the merge degrades to guesswork. This is the one genuinely load-bearing implementation requirement in the spec.
  • (B)/(C)/(D) are reasonable as opt-in flag behaviors, but a silent default winner contradicts the README's "will not overwrite without permission" guarantee.

3. Backend Scope for v1

Which sync backend(s) does reconciliation cover in this spec?

Current best-practice context: The two backends have different conflict models: git is branch divergence / fast-forward failure; hosted is an optimistic-concurrency version/hash 409. A reconciliation engine written against the shared Manifest/ManifestDiff types can be reused, but wiring, tests, and UX differ per backend, so covering both roughly doubles scope.
Recommended answer(s): (A)

Why these are recommended:

  • (A) is the smallest complete slice: git sync is the primary, fully-local-testable path and already has workspace diff to build on.
  • Design the core merge on the shared Manifest types so hosted reconciliation is a follow-up spec that reuses it — avoids over-building before the model is proven.
  • (C) is valid if you want parity now, but it enlarges the task breakdown and the proof surface significantly; call it out explicitly if that's the intent.

4. Command Surface + Review-Before-Write

How is reconciliation invoked, and does it produce a reviewable result before mutating the local manifest?

Current best-practice context: DevSpace's established pattern is "review then act" (plan writes last-plan.json; apply executes only the reviewed plan with a hash guard). A reconcile that shows the merged result before overwriting the local manifest fits that pattern and gives a natural rollback point.
Recommended answer(s): (A)

Why these are recommended:

  • (A) matches the repo's plan/apply idiom and its non-destructive posture: the user sees the reconciled manifest (and any flagged conflicts) before anything is written, with a hash guard on apply.
  • (B) is an acceptable lighter alternative if you want fewer commands, but in-place merge without a review artifact loses the rollback/inspection step.
  • (C) risks surprising mutations during a routine pull; (D) ships no actual reconciliation, just overwrite — it's the fallback, not the feature.
  • Recommend the reconcile step also back up the pre-reconcile local manifest (e.g. under app home) so the operation is reversible.
