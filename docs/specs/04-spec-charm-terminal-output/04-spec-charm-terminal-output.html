Spec

04-spec-charm-terminal-output

Sequence 04 · Feature: charm-terminal-output · Phase 1

Introduction/Overview

Today the devspace CLI prints plain, unstyled text: no color, no visual hierarchy, bare stdin confirmations, and machine-readable JSON on only two commands. This feature overhauls the terminal experience of every existing command using the Charm v2 ecosystem (charm.land): a shared lipgloss theme for all human-readable output, fang-styled help/errors at the root, leveled charm-log diagnostics for long-running commands, huh-based confirmations, and spinners for slow operations — while expanding --json coverage so scripts get a stable interface and keeping piped/NO_COLOR output free of ANSI escape codes. The goal is a CLI that is dramatically easier to scan on a TTY and strictly friendlier to scripts than it is today.

A full-screen interactive TUI dashboard is intentionally not part of this spec; it is the next spec in the sequence (05) and builds on the theme foundation established here.

Goals

  • Every user-facing command renders its human-readable output through one shared visual theme (severity colors, headers, tables) when stdout is a terminal.
  • Piped, redirected, NO_COLOR, and --no-color runs emit clean plain text containing zero ANSI escape sequences.
  • status, doctor, workspace diff, and mount --preview gain --json output with stable field names, joining the existing plan --json and setup plan --json as the documented scripting interface.
  • Confirmations and long-running operations (setup run/apply, project hydrate, scan) get modern interactive treatment with non-interactive fallbacks that keep scripted flows working.
  • make verify stays green: every output-related test assertion is deliberately updated, and plan/apply safety semantics are unchanged.

User Stories

  • As a developer, I want to see devspace status and devspace doctor with color-coded severity and aligned layout so that I can spot unhydrated, dirty, or broken projects at a glance.
  • As a developer, I want styled --help, usage, and error messages so that the CLI feels polished and commands are easier to discover.
  • As a script author, I want to run devspace status --json, doctor --json, workspace diff --json, and mount --preview --json so that I can parse results without scraping human-oriented text.
  • As a developer whose CI pipes devspace output to a log, I want plain deterministic text with no escape codes so that logs stay readable and diffs stay stable.
  • As a developer running devspace setup run, I want a clear interactive confirmation that still defaults to "no" so that I never run project-defined commands by accident.
  • As a developer hydrating a project, I want a spinner with a status line during the clone so that I know the operation is progressing rather than hung.

Demoable Units of Work

Three end-to-end vertical slices. Each is independently demoable on a real terminal and in a piped/CI context.

Unit 1: Theme Foundation and Root Command Polish

Purpose: Establish the one styling/detection layer every later change builds on, and make the CLI's first impression (help, errors, version) styled — without touching individual command output yet.

Functional Requirements:

  • The system shall add the Charm v2 dependencies to go.mod using charm.land/.../v2 import paths exclusively (lipgloss/v2, log/v2, fang/v2, huh/v2, bubbles/v2 + bubbletea/v2 for the spinner, plus github.com/charmbracelet/colorprofile); v1 github.com/charmbracelet/... paths shall not appear anywhere in the module.
  • The system shall provide a shared theme in a new internal/devspace/styles.go: named lipgloss styles for severity (ok/warn/fail/info), section headers, emphasis, muted text, and table borders, with light/dark-adaptive colors.
  • The system shall detect terminal capability once per invocation via colorprofile.Detect on the output writer and environment, and route all styled output through a colorprofile-aware writer so ANSI is automatically downsampled or stripped (piped output = plain text).
  • The system shall honor NO_COLOR, CLICOLOR, and CLICOLOR_FORCE, and shall add a persistent root flag --no-color that forces plain output even on a TTY.
  • The system shall wrap the root command with fang.Execute in cmd/devspace/main.go, providing styled help/usage/errors, automatic --version from build info, and manpage + completion subcommands; the existing version subcommand shall remain as an alias reporting the same version string.
  • The system shall route diagnostics of long-running surfaces (watch, hosted serve, mount) through charm.land/log/v2 to stderr with leveled, timestamped output (plain logfmt when stderr is not a TTY).

Proof Artifacts:

  • Screenshot: styled devspace --help and a styled error (unknown command) demonstrates fang integration.
  • CLI: devspace --help | cat and NO_COLOR=1 devspace --help transcripts containing zero ESC[ sequences demonstrates plain-output compliance.
  • Test: a unit test asserting styled-vs-plain rendering of themed helpers against TTY-like and buffer writers demonstrates the detection layer works.

Unit 2: Redesigned Command Output and Expanded --json

Purpose: Move every command's human output onto the shared theme with genuinely better layout (this is the visible payoff), and give scripts a stable JSON interface where they previously had to scrape text.

Functional Requirements:

  • The system shall restyle the output of status, doctor, plan, apply, workspace diff, env list, env recipient list, setup plan, and mount --preview using the shared theme: section headers, aligned key/value groups, and bordered tables where rows are homogeneous (recipient list, setup plan, mount preview).
  • The system shall keep doctor severity semantics visually distinct (OK/WARN/FAIL/INFO badges); wording and layout may change, but every existing check shall remain individually identifiable and the summary shall still state the hard-failure count.
  • The system shall migrate the raw fmt.Fprintf user-facing call sites in commands.go, mount.go, watch.go, doctor.go, and secrets.go onto helpers in output.go; helpers shall keep their io.Writer-based signatures so they stay unit-testable with buffers.
  • The system shall add a --json flag to status, doctor, workspace diff, and mount --preview, emitting via the existing writePrettyJSON (jsonio.go) with stable, documented field names and no ANSI content.
  • The system shall deliberately update every affected output assertion in commands_test.go, doctor_test.go, devspace_test.go, workspace_sync_test.go, mount_test.go, and mount_fuse_test.go, keeping assertion coverage equivalent: every fact asserted before (counts, names, severities, reasons) is still asserted after, against the new wording or the new JSON output.

Proof Artifacts:

  • Screenshot: side-by-side styled status, doctor, plan, and workspace diff on a real terminal demonstrates the redesigned layouts.
  • CLI: devspace status --json | python3 -m json.tool (same for doctor, workspace diff, mount --preview) demonstrates the machine-readable interface parses.
  • vhs GIF: a recorded tour of the restyled status/doctor/plan output (tape checked in under 04-proofs/) demonstrates the new look in motion.
  • CLI: devspace status | cat transcript with zero ANSI sequences demonstrates piped output stays plain.
  • Test: make verify passing demonstrates the redesign is fully covered by the updated suite.

Unit 3: Interactive Prompts and Progress

Purpose: Replace bare stdin prompts with modern interactive equivalents and show progress during slow operations, without weakening any safety gate or breaking scripted/CI usage.

Functional Requirements:

  • The system shall replace confirmSetup's raw prompt with huh/v2 forms that preserve current safety semantics: setup run uses a Confirm defaulting to "No"; setup apply (batch) keeps the typed-phrase gate via a huh Input whose validation requires the exact current phrase.
  • The system shall preserve the --yes bypass, and when stdin or stdout is not a TTY it shall fall back to the current plain bufio stdin behavior so piped flows (e.g. echo ... | devspace setup apply) keep working byte-for-byte.
  • The system shall display a bubbles/v2 spinner with a status line during project hydrate (git clone) and scan when stdout is a TTY, and plain start/finish lines when it is not.
  • The system shall keep secretInput's x/term.ReadPassword path unchanged for secret entry (no styling of secret prompts, no echo).

Proof Artifacts:

  • vhs GIF: setup run huh confirmation and the hydrate spinner recorded with vhs (tapes checked in under 04-proofs/) demonstrates the interactive flows.
  • CLI: piped confirmation transcript (non-TTY fallback) demonstrates scripted setup flows are unbroken.
  • Test: unit tests for the confirm fallback and spinner-off paths against buffer writers demonstrate non-TTY behavior.

Non-Goals (Out of Scope)

Clearly out of scope for this spec to manage expectations and prevent scope creep.

  1. TUI dashboard mode: the full-screen bubbletea application (project table, live watch pane, safe actions: scan, hydrate, plan, apply-safe) is deferred to spec 05, which builds on this spec's theme foundation. Nothing in this spec adds an alternate screen or event-loop UI.
  2. Markdown rendering (glamour): no current command renders markdown; not added.
  3. Hosted server behavior: the hardened HTTP contract (hardening_test.go) is untouched; only the hosted serve startup/log lines are restyled.
  4. FUSE/mount internals: only printed output changes; mounting behavior and guards stay as-is.
  5. Safety and path invariants: no change to plan/apply non-destructive semantics, Safety tagging, safeWorkspacePath, idempotency of init/scan, or secrets encryption/permissions.
  6. Sync/schema changes: no manifest schema bump, no new sync behavior.

Design Considerations

Follow the "rich TTY, honest pipe" pattern used by mature CLIs (e.g. gh): the same command renders styled layout on a terminal and clean plain text when piped. One adaptive palette lives in styles.go and is derived from the Liatrio brand palette (values verified from liatrio.ai stylesheets and logo assets):

  • Accent / headers / ok: Liatrio green, adaptive pair #89DF00 on dark terminal backgrounds / #24AE1D on light backgrounds (the two core greens from the Liatrio logo mark).
  • fail: #EF4343 (Liatrio's --destructive design token).
  • warn: Liatrio defines no warning token; use a complementary amber (#D97706 light / #FBBF24 dark), tunable at review.
  • info / muted: Liatrio neutral slates (#24292E foreground on light, muted grays from the #1A1F23 dark-theme family); the brand's yellow-green #C6F135 and purple #C068F9 accents are reserved for occasional highlights (e.g. table headers or badges), not severity.

Colors must be defined as light/dark adaptive pairs (lipgloss v2 explicit background detection) so they read on both terminal themes; all commands must look coherent side by side (same header treatment, same table borders, same badge shapes). Doctor's badges replace today's [OK]/[WARN]/[FAIL]/[INFO] prefixes visually but keep the same four severities. The styled gh pr status / gh repo view look is the reference aesthetic, rendered in Liatrio brand colors.

Repository Standards

Implementation must follow the repository's established patterns:

  • Single package internal/devspace with a thin cmd/devspace/main.go entrypoint; command wiring stays in commands.go, presentation helpers in output.go.
  • Output helpers take an io.Writer and a domain value and are unit-tested with buffers (see commands_test.go).
  • Tests isolate state with t.Setenv("DEVSPACE_HOME", t.TempDir()) and temp workspaces; never touch the real ~/.devspace.
  • make verify (test + vet + lint + build) is the local gate; CI runs go test, go vet, and build on every PR; make vulncheck runs in CI and will scan the new dependencies.
  • Conventional-commit style messages (feat:, docs:, chore:) as in recent history.

Technical Considerations

Grounded in latest-standards research (verified against charmbracelet repos, July 2026):

  • Versions: the Charm stack is GA on v2 — bubbletea v2.0.8, lipgloss v2.0.5, bubbles v2.1.0, log v2.0.0, huh v2.0.3, fang v2.0.1 — and interoperates cleanly (Go 1.25 directives; the module's Go 1.26 toolchain is fine). Use charm.land/.../v2 paths everywhere; v1 and v2 types do not interoperate.
  • Color detection: lipgloss v2 removed the implicit global renderer; capability detection is explicit via github.com/charmbracelet/colorprofile (Detect, NewWriter). This is what makes the plain-when-piped requirement natural: test buffers are non-TTY, so helpers render plain text in tests by default.
  • fang: depends only on lipgloss + cobra (no TUI stack), so adoption is one wrapper call at the entrypoint and trivially reversible. Its README still says "experimental" despite the stable v2 tag — accepted risk with a documented backout (revert main.go to plain Execute).
  • charm log: exposes an slog.Handler; long-running surfaces log through it to stderr, human-styled on TTY and logfmt otherwise.
  • huh: the recommended prompt layer; forms run standalone (no bubbletea program needed) and support a non-TTY error path we catch to trigger the bufio fallback.
  • JSON: reuse writePrettyJSON and existing domain structs where their JSON tags are already stable (Plan, SetupPlan); define small dedicated report structs for status/doctor/diff/mount preview rather than serializing internal state directly.

Security Considerations

  • Secret values are never styled, logged, or echoed: env list keeps masking, secretInput keeps x/term.ReadPassword, and charm-log calls must never receive env values as fields.
  • Generated .env files remain 0600; no change to age encryption or secrets storage.
  • Proof artifacts (transcripts, screenshots, JSON samples) must be produced in throwaway temp workspaces and contain no real tokens, remotes, or credentials.
  • New third-party dependencies are pinned in go.mod/go.sum and covered by CI's govulncheck; fang adds no network or exec behavior.
  • The confirmation redesign must not weaken the safety gate: default remains "No", batch apply keeps its typed phrase, and --yes remains the only bypass.

Success Metrics

  1. Theme coverage: 100% of user-facing human-readable output flows through output.go/styles.go helpers; no raw user-facing fmt.Fprintf remains in commands.go, mount.go, watch.go, doctor.go, secrets.go.
  2. Plain-output compliance: automated test proves piped and NO_COLOR output of every restyled command contains zero 0x1b escape bytes.
  3. Script interface: status --json, doctor --json, workspace diff --json, and mount --preview --json parse with a standard JSON parser in CI.
  4. Verification gate: make verify green with all output assertions deliberately migrated (coverage equivalent to the current ~68 assertions).
  5. Demo readiness: styled screenshots, transcripts, and vhs-recorded GIFs (with their .tape sources) captured under docs/specs/04-spec-charm-terminal-output/04-proofs/ suitable for the capstone demo.
