Spec

03-spec-fuse-lazy-mount

Sequence 03 · Feature: fuse-lazy-mount · Phase 1

Introduction/Overview

This feature resolves the remaining Plan 015 blocker by proving whether GitHub-hosted Linux runners can execute real FUSE mounts for the existing DevSpace lazy mount prototype. If hosted Linux is viable, the feature adds an opt-in integration job and FUSE-tagged test coverage for mount traversal, hydration-on-lookup, failure propagation, and clean unmounting.

Goals

  • Produce observed GitHub Actions evidence for Linux FUSE GO or NO-GO status.
  • Keep normal local and CI verification FUSE-free.
  • On GO, add an opt-in Linux FUSE integration job with real mount behavior coverage.
  • Update plan, design, proof, and validation artifacts so Plan 015 is no longer blocked.

User Stories

  • As a DevSpace maintainer, I want to know whether hosted CI supports the FUSE prototype so that I can decide whether to automate real mount testing.
  • As a CLI contributor, I want to run FUSE tests only on capable Linux hosts so that normal development remains portable.
  • As a reviewer, I want to see the workflow run, commands, and proof artifacts so that the Plan 015 outcome is evidence-based.

Demoable Units of Work

Each unit is a thin, verifiable slice of the Plan 015 decision path.

Unit 1: Hosted Linux FUSE Probe

Purpose: Determine whether GitHub-hosted Linux can run the real FUSE path.

Functional Requirements:

  • The system shall create a temporary probe workflow that checks /dev/fuse, fusermount3, a minimal go-fuse/v2 mount, directory listing, file read, and unmount.
  • The system shall capture the probe run URL, runner image details, and pass/fail output in proof artifacts.
  • The system shall remove the temporary probe workflow before final branch state.

Proof Artifacts:

  • GitHub Actions run URL and log excerpt demonstrate observed hosted-runner behavior.
  • Git status evidence demonstrates no temporary probe workflow remains.

Unit 2: GO or NO-GO Documentation

Purpose: Convert the probe result into durable project guidance.

Functional Requirements:

  • The system shall update docs/fuse-lazy-mount.md with a verified CI feasibility section.
  • The system shall update plans/README.md with a final Plan 015 status and one-line rationale.
  • The system shall update SDD-HTML proofs and validation with the final evidence.

Proof Artifacts:

  • Documentation diff demonstrates the GO or NO-GO decision and supporting evidence.
  • Plan table excerpt demonstrates the Plan 015 status is reconciled.

Unit 3: Linux FUSE Integration Job On GO

Purpose: Add real mount test coverage only when hosted Linux has proven FUSE support.

Functional Requirements:

  • The system shall add a Linux-only mount-integration job only if the probe is GO.
  • The system shall add a build-tagged mount_fuse_test.go that verifies traversal before hydration, hydration-on-lookup from a local bare Git remote, failed hydration propagation, and clean unmount.
  • The system shall keep make verify and the default CI verify job FUSE-free.

Proof Artifacts:

  • CI job output demonstrates go test ./internal/devspace -run Fuse -tags fusetest -v passed on a FUSE-capable runner.
  • make verify output demonstrates the normal gate remains FUSE-free.

Non-Goals (Out of Scope)

  1. macOS FUSE CI: macOS runner FUSE support remains a future evaluation.
  2. Product behavior changes: The existing devspace mount behavior should not change except for testability fixes required to validate documented behavior.
  3. Richer status cards: dirty repo, missing .env, and setup hint display remain follow-up work.
  4. Path-vs-name mount design: Long-term virtual directory decisions remain out of scope.

Design Considerations

No specific UI design requirements identified. HTML artifacts should be reviewer-friendly and self-contained.

Repository Standards

  • Follow Go formatting and keep tests beside implementation in internal/devspace/.
  • Use make verify as the normal local CI gate.
  • Keep generated build outputs out of source and avoid committing secrets or workspace state.
  • Use Conventional Commit style for implementation commits.
  • Preserve the repository instruction that default CLI flows must work without FUSE.

Technical Considerations

The existing prototype uses github.com/hanwen/go-fuse/v2/fs. GitHub-hosted runner software changes over time, so the implementation must rely on observed workflow evidence rather than local inference. Build tags such as linux && fusetest keep FUSE tests opt-in and portable.

Security Considerations

  • Proof artifacts must not include GitHub tokens, hosted tokens, age identities, or real environment values.
  • The probe workflow must not require repository secrets.
  • FUSE tests must use temporary directories and local bare Git remotes only.

Success Metrics

  1. Probe outcome: Linux hosted runner status is documented as GO or NO-GO with an observed run.
  2. Default verification: make verify exits 0 without FUSE.
  3. Plan reconciliation: Plan 015 is no longer BLOCKED without evidence.
  4. Review gate: coderabbit review --agent is run after implementation and still-valid findings are addressed.

Open Questions

No blocking open questions. Assumption: the user authorized temporary branch push and GitHub Actions execution by requesting implementation of this plan.
