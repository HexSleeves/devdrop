Task 03 Proofs - Hosted-backend reconciliation resolves HTTP 409 version conflicts

Sequence 06 · Feature: manifest-reconciliation · Task 3.0 · Phase 3

Task Summary

This task extended reconciliation to the hosted control-plane backend. When a hosted push is rejected with the optimistic-concurrency version conflict, devspace hosted reconcile now fetches the current hosted manifest with its version/hash baseline, runs the same shared three-way engine against the Task 1.0 base snapshot, and writes the same review-first plan artifact (tagged backend: hosted). On --apply it pushes the merged manifest to the server using the fetched version as the concurrency baseline — so the server accepts it, and a racing third push still safely fails — then writes the merged manifest locally and refreshes the hosted sync state and base snapshot. Conflict handling, backup, and hash guard are byte-for-byte the same code paths as the git backend.

What This Task Proves

  • A real version-conflict scenario (two machines pushing divergent manifests to one hosted server) is resolved by reconcile into the union, accepted by the server, with the client baseline updated.
  • A genuine conflict blocks the hosted push — neither the server manifest nor the local manifest changes — until an explicit force choice is given.
  • A successful hosted reconcile refreshes the base snapshot (Unit 1 integration), keeping future three-way merges accurate.
  • The engine, plan artifact, force semantics, backup, and guard are shared with the git backend — the hosted wiring is thin (~140 lines plus the command).

Artifact: Push-first apply sequence with version-conflict protection

What it proves: Apply pushes the normalized merged manifest with client.put(ctx, fetchedVersion, …) before any local write; if another machine pushed between the reconcile's GET and the apply's PUT, the server rejects it and nothing changes locally.

Why it matters: Ordering is the safety property here: writing locally before the server accepted the merge could leave local state claiming a sync that never happened. The push-first sequence plus the local hash guard makes the apply atomic-in-effect from the user's perspective.

Reconcile computes a three-way merge between the last synced base manifest, the local
manifest, and the hosted backend manifest.

By default it writes only a review artifact to DEVSPACE_HOME/last-reconcile.json and does
not change the workspace manifest or hosted backend.

With --apply, DevSpace writes DEVSPACE_HOME/manifest-backup.json, pushes the merged
manifest to the hosted backend with version-conflict protection, then replaces the local
manifest and refreshes the hosted sync baseline. Apply is guarded by the local manifest
hash captured when reconcile was generated.
